# Requirements

This is the canonical, numbered register of every requirement in Orbit, organised by phase.

It exists so Jira Epics and Stories can reference a stable ID instead of restating requirements in prose — e.g. an Epic description can say *"This epic covers `P1-09`–`P1-12` of the Phase 1 requirements documented in `docs/REQUIREMENTS.md`."*

**This document is the source of truth for requirement IDs.** IDs are phase-scoped and should be treated as stable once assigned — do not renumber existing IDs, even if a requirement is later reworded. If a requirement is removed, leave a gap rather than reusing its ID for something else.

Each ID also appears inline next to its bullet in `FEATURES.md`, for full context and rationale.

**Descriptions are a snapshot at time of writing — for authoritative schema, see `erd.md`, and for endpoint contracts, see `API_CONTRACTS.md`.**

## Phase 1 — Core

| ID    | Category        | Requirement                                                | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
|-------|-----------------|------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| P1-01 | Authentication  | User registration with email and password                  | A new user creates an account with an email, password, and display name. The email must be unique and the password must meet a minimum strength bar. On success, the user is signed in immediately — no separate login step required.                                                                                                                                                                                                                                                                                                                                                                           |
| P1-02 | Authentication  | User login with email and password                         | An existing user authenticates with their email and password and, on success, is signed in immediately.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| P1-03 | Authentication  | JWT-based auth — access token 15 min, refresh token 7 days | Every request from a signed-in user is checked once at the door; once confirmed, that identity is trusted for the rest of the request. The active session is short-lived by design (15 minutes) to limit exposure if it's ever compromised, while a longer-lived renewal window (7 days) lets the user stay signed in without repeatedly logging back in — each renewal replaces the previous one so an old one can't be reused. A real-time connection is checked once when it's first opened and stays trusted for as long as it's open, even if the underlying session would otherwise have expired by then. |
| P1-04 | Authentication  | Protected routes on the frontend                           | Pages that require a signed-in user are inaccessible to anyone who isn't logged in — they're redirected to the login page instead, and returned to the page they originally wanted after logging in.                                                                                                                                                                                                                                                                                                                                                                                                            |
| P1-05 | Authentication  | Logout with token invalidation                             | A user can end their session on demand. Doing so invalidates their current session so it can't be reused, and cleanly disconnects any active real-time connection — the user is marked offline and that becomes visible to their contacts.                                                                                                                                                                                                                                                                                                                                                                      |
| P1-06 | User Profile    | Display name                                               | Every user has a display name shown throughout the app — on their profile, in message history, and anywhere else they're identified. If a user later changes their name, messages they already sent continue to show the name as it was at the time.                                                                                                                                                                                                                                                                                                                                                            |
| P1-07 | User Profile    | Bio (short text)                                           | A user can add a short bio describing themselves, shown on their profile.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| P1-08 | User Profile    | Profile visible to other users                             | Any authenticated user can view another user's basic profile (display name, bio, avatar). If a BLOCKED relationship exists between viewer and target, the response indicates the profile is unavailable rather than exposing the underlying reason. Full behavior: discussions/007_blocking_behavior.md.                                                                                                                                                                                                                                                                                                        |
| P1-09 | Contacts        | Send a connection request                                  | A user can send a request to connect with another user. Sending a duplicate request to the same person, or to someone they're already connected with, is prevented.                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| P1-10 | Contacts        | Accept or decline an incoming connection request           | A user can respond to an incoming connection request. Accepting establishes the connection and immediately creates a ready-to-use direct conversation between the two users, with both notified. Declining is quiet and non-restrictive — the requester isn't notified, and nothing prevents them from sending a new request later. Full reasoning: discussions/008_connection_request_decline_strategy.md.                                                                                                                                                                                                     |
| P1-11 | Contacts        | View contact list                                          | A user can see everyone they're connected with, along with each contact's basic profile info and whether they're currently online.                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| P1-12 | Contacts        | Block a user                                               | Blocking is one-directional in its effects: the blocker retains full visibility and normal messaging ability toward the blocked user, while the blocked user loses presence visibility, sees a masked display name, cannot view the blocker's profile, and has messages silently undelivered. Existing chat history is preserved for both. Group messaging between the two is unaffected. Full behavior: discussions/007_blocking_behavior.md.                                                                                                                                                                  |
| P1-13 | Direct Messages | Real-time message delivery via WebSocket                   | Messages sent in a direct conversation reach the recipient instantly while they're active in the app. A message is always saved permanently before it's delivered live, so a delivery hiccup never risks losing it — at worst it just shows up a little later once the recipient's history refreshes.                                                                                                                                                                                                                                                                                                           |
| P1-14 | Direct Messages | Message history on open — last 50, paginated on scroll     | Opening a conversation loads its most recent messages first. Scrolling further back loads older messages progressively rather than pulling in the entire history at once.                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| P1-15 | Direct Messages | Edit your own message                                      | A user can edit the text of a message they sent, and the change is reflected for the other participant right away. Only their own plain-text messages can be edited — not images or files.                                                                                                                                                                                                                                                                                                                                                                                                                      |
| P1-16 | Direct Messages | Delete your own message                                    | A user can delete a message they sent. It's replaced with a placeholder for both participants indicating it was deleted, while the surrounding conversation stays intact.                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| P1-17 | Direct Messages | Typing indicator                                           | While a user is actively typing, the other participant sees a live typing indicator, which clears shortly after typing stops (or automatically after a short pause, as a safety net).                                                                                                                                                                                                                                                                                                                                                                                                                           |
| P1-18 | Direct Messages | Read receipts (delivered / seen)                           | The sender can see their message's status move through sent → delivered → read, shown as a simple visual indicator. Once read, the status never moves backward.                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| P1-19 | Direct Messages | Online/offline presence per contact                        | A user can see whether each of their contacts is currently online, updated live. Having several tabs open at once still counts as a single "online" — presence doesn't flicker as tabs open and close.                                                                                                                                                                                                                                                                                                                                                                                                          |
| P1-20 | Direct Messages | Server-side unread count (no live push yet)                | Each conversation shows how many messages the user hasn't yet seen. This count is accurate whenever the app is loaded or reopened, though it doesn't yet update live while the app is already open — that arrives in a later phase.                                                                                                                                                                                                                                                                                                                                                                             |
| P1-21 | Groups          | Create a group with name, description, topic tag           | A user can create a group with a name, description, and topic tag, choosing whether it's public or private, and automatically becomes its first admin.                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| P1-22 | Groups          | Public groups — discoverable and joinable                  | Public groups can be found by any user browsing or searching, and joined directly with no invitation needed.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| P1-23 | Groups          | Private groups — invite only                               | Private groups can't be discovered by browsing — the only way in is an invite link generated by a group admin, which expires after a set period.                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| P1-24 | Groups          | Group member list with roles (Admin, Member)               | Any group member can view the full member list and see who holds which role — admin or regular member.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| P1-25 | Groups          | Leave a group                                              | A member can leave a group at any time. If they were the group's only admin, another member is automatically promoted so the group keeps functioning; if they were the last member, the group is closed.                                                                                                                                                                                                                                                                                                                                                                                                        |
| P1-26 | Groups          | Admin can remove a member                                  | A group admin can remove any regular member from the group, but cannot remove another admin.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| P1-27 | Groups          | Admin can delete the group                                 | A group admin can permanently delete the group, its entire message history, and its membership, with all members notified as it happens.                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| P1-28 | Group Messaging | Real-time message delivery via WebSocket                   | Same instant, save-first delivery behavior as direct messages (P1-13), extended to every member of the group.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| P1-29 | Group Messaging | Message history on open — last 50, paginated               | Same progressive history loading as direct messages (P1-14), applied to group conversations.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| P1-30 | Group Messaging | Edit your own message                                      | Same editing rules as direct messages (P1-15) — a user can edit their own plain-text messages within the group.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| P1-31 | Group Messaging | Delete your own message                                    | Same delete behavior as direct messages (P1-16), with one addition: a group admin can also remove any member's message, not just their own.                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| P1-32 | Group Messaging | Typing indicator ("3 people are typing...")                | Same live typing indicator as direct messages (P1-17), extended to show how many members are typing at once when more than one person is.                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| P1-33 | Group Messaging | Online/offline presence per group member                   | Same live online/offline visibility as P1-19, applied to every member of a group rather than just direct contacts.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| P1-34 | Group Discovery | Browse public groups                                       | A user can browse the full list of public groups without needing to search for anything specific.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| P1-35 | Group Discovery | Filter groups by topic tag                                 | The group browsing list can be narrowed down to groups matching a chosen topic tag.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| P1-36 | Group Discovery | Join a public group directly from discovery                | A user can join a public group directly from the browsing view, with no extra steps beyond that.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |

## Phase 2 — Enhancement

| ID    | Category                 | Requirement                                        | Description                                                                                                                                                                                                                                                                                                                                                                                                                               |
|-------|--------------------------|----------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| P2-01 | Message Interactions     | Emoji reactions on any message                     | A user can react to any message with an emoji. Reacting with the same emoji again removes it; reacting with a different emoji swaps it for the new one — each person can have only one reaction on a given message at a time, though different people can react with different emoji.                                                                                                                                                     |
| P2-02 | Message Interactions     | Quoted reply with original shown inline            | A user can reply to a specific message, with the original message's content shown inline above their reply for context. The quoted content is captured at the moment of the reply, so it stays visible and unchanged even if the original message is later edited or deleted.                                                                                                                                                             |
| P2-03 | Notifications            | Live unread badge per conversation (Kafka → queue) | Builds on the unread count already computed in Phase 1: instead of only being accurate on load or reopen, the badge now updates live for users who already have the app open, the same way messages and presence already update in real time. This behaves the same for every conversation, whether or not it's muted.                                                                                                                    |
| P2-04 | Notifications            | Browser desktop notifications when tab unfocused   | When a user's browser tab isn't focused, a new message triggers a native desktop notification via the browser. This is handled entirely on the client using the browser's own notification permission. If the conversation is muted, this notification is skipped — muting affects only this popup, not the unread badge.                                                                                                                 |
| P2-05 | User Profile Updates     | Upload and update profile avatar (R2)              | A user can upload an image as their profile picture, replacing any existing one. Only common image formats are accepted, up to a set file size limit, and the previous avatar is removed once the new one is in place.                                                                                                                                                                                                                    |
| P2-06 | User Profile Updates     | Update display name and bio                        | A user can change their display name and bio at any time. As with Phase 1, changing the display name doesn't rewrite the sender name on messages already sent — those keep showing the name as it was when sent.                                                                                                                                                                                                                          |
| P2-07 | Group Management Updates | Upload group avatar and cover image (R2)           | A group admin can upload or replace the group's avatar and cover image, shown to all members. Unlike a personal avatar, this is an admin-only action, not something any member can do.                                                                                                                                                                                                                                                    |
| P2-08 | Group Management Updates | Pin an important message in a group                | A group admin can pin a message so it stays easily accessible to all members, and can unpin it later. Pinning captures the message's content at the time it was pinned, so it's still readable even if the original is later edited or deleted. A group can hold up to 10 pinned messages at once; once that limit is reached, an admin must unpin something before pinning anything new — the oldest pin is never removed automatically. |
| P2-09 | Group Management Updates | Mute a conversation (DM or group)                  | A user can mute a direct or group conversation so new messages in it no longer trigger a desktop notification, without affecting how it appears for anyone else in that conversation. Muting is personal and reversible at any time. It only silences the popup notification — the conversation's unread badge still updates normally, so the user can still see at a glance that something arrived.                                      |

## Phase 3 — Search & Files

| ID    | Category             | Requirement                               | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|-------|----------------------|-------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| P3-01 | Search               | Search users by display name              | A user can search for other users by typing part of their display name, with a short minimum length required before results appear. Each result shows the matched user's basic profile info along with the searcher's current connection status with them (not connected, request sent, request received, connected, or blocked). Anyone the searcher has blocked, or been blocked by, is left out of the results entirely.                                                                                                                                                                                                                                                |
| P3-02 | Search               | Search public groups by name or topic tag | A user can browse or search public groups by typing part of the group's name or description, and can further narrow results down to a specific topic tag — the two can be used together or on their own. Browsing and searching public groups works even without being signed in; joining a group or viewing its full details still requires an account. Private groups never appear here, consistent with them being invite-only and undiscoverable.                                                                                                                                                                                                                      |
| P3-03 | Search               | Search messages within a conversation     | A user can search for matching text inside a single conversation they're already part of, using a short minimum-length query, and see the matching message's content, sender, and time sent. This search is scoped to one conversation at a time — it does not search across every conversation the user is in. If the user was previously blocked by the other participant and kept sending messages during that time, their own messages from that period still turn up when they search their own history — the blocking participant still never sees them, consistent with how blocking already works elsewhere. Full reasoning: discussions/007_blocking_behavior.md. |
| P3-04 | File & Image Sharing | Send images inline in a message           | A user can attach an image from their device to send as a message instead of typing text. Common image formats are accepted, up to a set size limit. Once sent, the image behaves like any other message for deleting, reacting, and replying — the one difference is that, unlike text messages, an image can't be edited afterward.                                                                                                                                                                                                                                                                                                                                      |
| P3-05 | File & Image Sharing | Inline image preview in the conversation  | An image sent in a conversation is shown directly inside the chat as a visual preview, rather than requiring the recipient to download it or open a separate viewer. Since the underlying image link is temporary for security, the app quietly fetches a fresh one behind the scenes whenever needed, so the preview keeps working without the user ever noticing a link expired. If the message is later deleted, the image is permanently removed rather than just hidden, so the preview cannot be brought back afterward.                                                                                                                                             |
| P3-06 | File & Image Sharing | Send non-image files as attachments       | A user can attach any other kind of file — documents, PDFs, and similar — that isn't an image, up to a larger size limit than images since these aren't shown as a visual preview. The attachment appears in the conversation as a distinct file card rather than inline artwork.                                                                                                                                                                                                                                                                                                                                                                                          |
| P3-07 | File & Image Sharing | Download link for file attachments        | Every file attachment includes a link the recipient can use to download it. Because the underlying storage link is temporary, a fresh one is generated automatically the moment it's needed, so the download keeps working even long after the message was originally sent — unless the message itself has since been deleted, in which case the file is gone for good.                                                                                                                                                                                                                                                                                                    |
| P3-08 | File & Image Sharing | Files and images stored in Cloudflare R2  | Uploaded images and files are kept in the same external file storage already used for avatars and group images elsewhere in the app, rather than in the database itself. The upload to storage must succeed before the message is created at all — if the upload fails, no message is created, so a user never ends up with a message that looks sent but has nothing behind it. Deleting the message later permanently removes the stored file too, not just the message content. Full reasoning: discussions/011_message_attachment_cleanup_on_delete.md.                                                                                                                |

## Phase 4 — AI Features

| ID    | Category                | Requirement                                                  | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
|-------|-------------------------|--------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| P4-01 | AI Summary Bot          | `/summary` command in any group or DM                        | A user can request a catch-up of a conversation they're part of — DM or group — by issuing a command instead of typing a normal message. The command itself is not posted as a message; it's replaced entirely by the response. The summary is generated from a recent, bounded slice of history rather than the entire conversation, so it stays fast and stays within what the assistant can reasonably work with in one pass.                                                               |
| P4-02 | AI Summary Bot          | Returns a concise summary as a bot message                   | The generated summary is posted into the conversation itself, attributed to a dedicated AI assistant identity, and everyone in that conversation sees it exactly like any other message — including anyone who opens the conversation later. It isn't returned privately to the requester only. If generation fails for any reason, nothing is posted and the conversation is otherwise unaffected — the user can simply try again.                                                            |
| P4-03 | AI Assistant Bot        | `@ask` mention followed by a question in a group             | A member of a group can address a question to the AI assistant by mentioning it inline, the same way they'd address a person, followed by their question. Unlike the summary command, this doesn't replace the member's message — the question is sent and shown exactly like any other message in the group. Available in groups only; in a direct conversation, the same text is just sent as ordinary text with no special effect. Full reasoning: discussions/012_ask_mention_behavior.md. |
| P4-04 | AI Assistant Bot        | Bot responds inline, visible to all group members            | Once the question has been posted, the assistant's answer is generated and posted as its own separate message directly underneath it — from the same AI identity used for summaries — visible to every current and future member of the group, not returned privately to whoever asked. This makes it a shared, reusable answer rather than a one-off private lookup. If the assistant can't produce an answer, nothing is posted and the group conversation is unaffected.                    |
| P4-05 | Smart Reply Suggestions | 2-3 contextual quick replies after receiving a message       | Immediately after a user receives a new message, the app quietly prepares a small handful of quick-reply options and surfaces them near the message input — the user doesn't have to ask for them. These are private to the person seeing them; no one else in the conversation is aware they were offered or what they were.                                                                                                                                                                  |
| P4-06 | Smart Reply Suggestions | One click to send as-is or edit before sending               | Any suggested reply can be sent immediately with a single tap, or opened up and edited first if it's close but not quite right. Suggestions are a head start, not a forced response.                                                                                                                                                                                                                                                                                                           |
| P4-07 | Smart Reply Suggestions | Suggestions generated from conversation context, not generic | The suggestions reflect what was actually said — the specific message received and the recent thread around it — rather than being a fixed set of canned replies reused everywhere. Two different messages should reasonably produce two different sets of suggestions.                                                                                                                                                                                                                        |
| P4-08 | Message Translation     | Per-message "Translate to English" option                    | Any message a user can see, in any conversation, can be translated to English individually and on demand — this isn't an always-on setting, it's a per-message action the reader chooses to take.                                                                                                                                                                                                                                                                                              |
| P4-09 | Message Translation     | Translated text appears inline below original                | The translation shows up directly underneath the original message in place, rather than opening a separate view or replacing anything. It's visible only to the person who asked for it — other participants continue to see just the original.                                                                                                                                                                                                                                                |
| P4-10 | Message Translation     | Original message remains visible                             | Requesting a translation never hides, replaces, or alters the original message for anyone. Everyone else in the conversation continues to see it exactly as it was sent, in its original language.                                                                                                                                                                                                                                                                                             |
| P4-11 | Tone Checker            | Optional toggle in message input settings                    | Tone checking is opt-in and personal — a user turns it on for themselves in their own settings. With it off (the default), sending a message behaves exactly as it does everywhere else in the app, with no extra step.                                                                                                                                                                                                                                                                        |
| P4-12 | Tone Checker            | Analyses message before sending                              | Once enabled, pressing send doesn't immediately transmit the message — it's first checked, and only proceeds to actually send once that check completes (or is bypassed by the user, see P4-15).                                                                                                                                                                                                                                                                                               |
| P4-13 | Tone Checker            | Flags aggressive, unclear, or likely-misread messages        | The check specifically looks for tone problems — wording that could come across as aggressive, that's ambiguous, or that's likely to be misread by the recipient. If none of these apply, the message sends normally with no interruption at all.                                                                                                                                                                                                                                              |
| P4-14 | Tone Checker            | Suggests a softer or clearer alternative                     | When a message is flagged, the user isn't just told there's a problem — they're offered a rewritten version intended to address whatever was flagged, so they have something concrete to work with instead of starting over.                                                                                                                                                                                                                                                                   |
| P4-15 | Tone Checker            | User can proceed with original or use the suggestion         | The flag is advisory, not a block. The user decides: send their original wording anyway, or swap in the suggested rewrite instead — either way, the choice of what actually gets sent stays theirs.                                                                                                                                                                                                                                                                                            |
